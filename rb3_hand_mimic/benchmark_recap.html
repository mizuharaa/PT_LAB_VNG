What the numbers say

Architecture win

Control never stalls on detection

115 Hz control in both modes proves the detection↔control split: a slow or stalled tracker (or the not-yet-real DexH15 SDK) cannot drag down the command loop. Pose age stays low, so commands act on fresh data.

Bottleneck

Landmark inference on CPU

~60–70 ms/frame while tracking is the whole story for single-cam fps (~14). Everything downstream — mapping, fusion, safety — is sub-millisecond. To go faster you make inference cheaper, not the pipeline.

Dual fusion is cheap on latency

Two cameras, tighter tails

Running both cameras in parallel doubled frame throughput and actually lowered end-to-end p90/p99 (fusion picks the fresher of two views). The cost is CPU, not lag.

On the target device

RB3 will read differently

These are x86-CPU numbers. The RB3's aarch64 CPU is slower for this model unless a GPU/Hexagon delegate is used — expect lower fps until accelerated. Re-run the same benchmark on the RB3 to set the real baseline.

How to push it faster

  • Accelerate inference — the only lever that moves the ceiling: GPU/Hexagon (QNN) delegate for the Tasks model, or a lighter/quantized hand model.
  • Drop capture resolution — try 320×240 @ 60; landmark cost scales with input pixels.
  • Single camera if CPU-bound — --no-dual halves inference load when the second view isn't earning its keep.
  • Re-measure on the RB3 — python main.py --benchmark 30 (with a hand held in frame the whole time for true end-to-end numbers).
